## 本轮讨论目标
- 在既定 Web 薄入口方案下，收敛第一阶段 UI 界面的信息结构、页面区域、关键元素与功能覆盖范围。
- 明确哪些能力必须进入首版页面，哪些能力应延后，避免 UI 设计倒逼架构膨胀。
- 形成后续实施前可执行的页面蓝图与测试门禁，不直接改代码。

- 用户明确要求发起团队讨论，应按团队方式并行分析。

- 会话真相源仍为 `STM`，前端不得维护第二套独立历史状态。
- 第一阶段不引入前后端分离、WebSocket、流式推送、复杂富文本消息时间轴。

## Team 分工（架构/实现/测试/风险）
- 架构：收敛页面信息架构、主交互流和会话边界。
- 实现：评审首版页面应包含的具体区块、控件和状态反馈。
- 测试：定义 UI 层应覆盖的功能面、页面门禁和回归重点。
- 风险：识别首版 UI 若过度设计或漏掉关键控制项带来的风险。

## 共识结论（是否通过 + 条件）
- 结论：通过，可以将 Web 第一阶段 UI 收敛为“单页任务控制台”形态，而不是类聊天产品形态。
- 通过条件：
  - 页面以单页为主，围绕“发起一轮请求、查看结果、继续会话、重置会话”四个核心动作展开。
  - 页面结构采用“顶栏 + 主工作区 + 辅助状态区”三段式；桌面端以“两栏 + 顶栏”为主，移动端退化为纵向单栏。
  - 首版必须显式暴露 `goal`、`action_type`、`target`、`session_id`、当前状态、历史区、结果区、重置入口。
  - 会话真相源必须仍为 `STM`，前端只持有瞬时展示状态，不维护第二套会话历史。
  - 第一阶段不引入多会话列表、流式输出、富文本消息、复杂调试面板、前后端分离 SPA。
  - UI 门禁以关键流程可操作、关键状态可见、关键错误可识别为主，不做像素级视觉回归。

## 风险清单（高/中/低）
- 高：
  - 若首版按“聊天产品”方向扩张为多标签会话、富文本消息流、流式输出、复杂主题切换，会把 Web 薄入口升级成产品重构。
  - 若页面不显式暴露 `action_type`、`target`、`session_id`、状态与重置入口，Web 将丢失当前 CLI 的最小可控性。
  - 若前端维护独立历史真相源，而非以后端 `STM` 为准，会导致会话恢复、状态展示和测试断言失真。
- 中：
  - 若桌面端做复杂多栏布局，移动端会同时承受响应式、滚动冲突和可访问性压力。
  - 若没有加载态、错误态、空态、禁用态，用户会误判为请求未发送或系统卡死。
  - 若历史区、结果区、状态区不分离，`clarify`、成功响应和失败原因会混在一起，影响可用性。
  - 若前端直接拼复杂请求对象而非依赖稳定 API 契约，后续字段调整会引入大量联动回归。
- 低：
  - 主题切换、快捷命令面板、个性化皮肤等增强项若延后，整体风险可控。
  - 帮助提示与字段说明暂时简化会提高学习成本，但不会破坏主链能力。
  - 前端保留少量瞬时 UI 状态，如输入中、按钮 loading、局部折叠，风险较低。

## 执行顺序（Phase 1..N）
- Phase 1：冻结首版页面信息架构
  - 顶栏：产品名、`session_id`、连接/状态标记、重置会话。
  - 左栏或上区：任务配置卡片、状态卡片、会话控制。
  - 右栏或下区：结果摘要区、会话历史区。
- Phase 2：冻结首版必备元素与功能覆盖
  - 输入：`goal`、`action_type`、条件化 `target`、发送按钮、清空输入。
  - 展示：结果摘要、历史列表、角色标签、响应类型标签。
  - 控制：刷新状态、复制 `session_id`、重置会话。
  - 状态：加载、空态、字段校验错误、后端错误、`clarify` 引导。
- Phase 3：冻结首版不做项
  - 多会话列表、多标签页、WebSocket、流式输出、富文本消息、复杂时间轴、前后端分离 SPA、复杂调试面板。
- Phase 4：冻结首版 UI 测试门禁
  - `基础提交流`
  - `会话续写流`
  - `clarify/错误映射流`
  - `重置流`
- Phase 5：待用户明确同意后，再进入实现讨论或代码修改阶段。
