# CLI 改为 Web 端团队评审

- 记录时间：2026-03-14
- 用户原始诉求：现在发起团队评审，抛弃 cli 方案，将 cli 方案修改为 web 端
- 当前状态：讨论中（未进入代码修改）。

## 本轮讨论目标
- 评审“抛弃 CLI、切换到 Web 端”是否可行，以及需要修改到什么范围。
- 明确当前 CLI 方案中哪些能力应保留为内核，哪些入口与测试需要迁移。
- 形成后续是否进入代码修改的执行顺序、风险和门禁条件。

## 当前已知约束
- 本轮属于项目问题，需先记录并进入讨论。
- 用户已明确要求发起团队评审，应按团队方式并行分析。
- 未获得用户对评审结论的明确同意前，不执行代码修改。
- 当前子仓 `sub/MzAgentSub` 现有入口以 CLI / 最小 REPL / demo 脚本为主，尚无 Web 端入口。

## Team 分工（架构/实现/测试/风险）
- 架构：评审 CLI 到 Web 的切换是否会牵动主链内核，判断合理的阶段切分。
- 实现：梳理最小迁移范围、必须新增的 Web 入口、需要废弃但不能直接删除的旧入口文件。
- 测试：判断哪些现有测试仍然有效，哪些入口层测试与 README 表述会失真，并给出 Web 最小测试矩阵。
- 风险：评估直接抛弃 CLI 的高/中/低风险，并判断 CLI 是否应先降级为内部调试入口。

## 共识结论（是否通过 + 条件）
- 结论：通过，可以进入“Web 端替代 CLI 主入口”的后续实施讨论，但当前**不通过**“直接物理抛弃 CLI 并立即删改现有入口”的做法。
- 通过条件：
  - 保留 `Pipeline / STM / adapters / contracts / runtime / perception` 作为内核，不把入口替换误做成架构重写。
  - 先从 `cli.py` 抽共享入口服务层，再新增 Web 薄入口。
  - 先让 Web 集成测试接管正式门禁，再决定是否把 CLI 降级或移入 `distroy/`。
  - README、测试名称、测试速查表和默认门禁必须同步改口。

## 风险清单（高/中/低）
- 高：若直接删除 CLI，会同时失去现有最小运行入口、手工调试入口与集成验证锚点。
- 高：`integration`、`REPL`、README 当前都把 CLI 当正式入口，不先替换会导致“可运行”证据断层。
- 高：若 Web 化时顺手引入完整服务端产品能力，会把入口迁移升级成架构重写。
- 中：CLI 的多轮 REPL 语义与 Web 会话语义并不等价，会话绑定、重连、并发隔离都需要重新定义。
- 中：当前输入模型依赖 `goal / action-type / target / stm-path` 等强约束，Web 若放宽输入会牵动 `clarify` 与目标指定策略。
- 中：Web 会引入 HTTP 服务、静态资源、启动方式和浏览器交互测试，运维面与依赖面上升。
- 低：核心主链、协议层、适配层和大部分测试资产可以继续复用。
- 低：CLI 保留为内部调试入口可以作为 Web 迁移期的稳定对照组。

## 执行顺序（Phase 1..N）
- Phase 1：从 `cli.py` 抽共享入口服务层，解耦参数解析与主链调用。
- Phase 2：新增 Web 薄入口，只先提供单轮请求、会话历史、重置与状态读取能力。
- Phase 3：迁移正式集成门禁：
  - 新增 Web roundtrip / session / clarify / error mapping 测试。
  - 让 Web 测试接管 `integration` 正式门禁。
- Phase 4：迁移 README、测试速查表、demo 说明与默认入口文案。
- Phase 5：待 Web 端具备等价证据后，再决定 CLI 是降级为内部调试入口，还是移动到 `distroy/`。
