## 本轮讨论目标
- 在“Web 端替代 CLI 主入口”的前提下，收敛当前仓库最合适的 Web 实现方案。
- 明确后端入口、会话模型、前端交互形态、测试门禁与迁移顺序。
- 形成可进入后续实施讨论的实现蓝图，不直接改代码。

  - 应先抽共享入口服务层，再新增 Web 薄入口

## Team 分工（架构/实现/测试/风险）
- 架构：收敛 Web 端最合适的薄入口结构、共享服务层切分与会话模型。
- 实现：评审后端最小可落地方案、推荐的 Python Web 形态、API 列表与新增目录文件。
- 测试：定义 Web 实施后必须新增的测试集合、旧测试与旧文档的降级方式、默认门禁约束。
- 风险：沿用上一轮关于“不可直接物理删除 CLI”的边界，约束本轮方案不膨胀成架构重写。

## 共识结论（是否通过 + 条件）
- 结论：通过，可以将“Web 端第一阶段实现方案”收敛为：
  - `轻量 ASGI JSON API 服务` 作为正式主入口
  - `单页薄界面 + 原生表单/轻量 JS` 作为第一阶段前端壳
  - 当前 CLI 降级为内部调试/兼容入口，而不是立即物理废弃
- 通过条件：
  - 必须先从 `cli.py` 抽共享入口服务层，不在 Web handler 中直接拼装 `Pipeline / AdapterHub / FileBackedSTM / ExecutionContext`
  - Web 第一阶段应以 API 优先，不引入前后端分离、WebSocket、流式推送、浏览器重型门禁
  - 会话真相源必须仍然是 `STM`，Web 不得再维护第二套历史状态
  - Web 集成测试必须接管正式入口门禁后，CLI 相关测试和文档才能降级

## 风险清单（高/中/低）
- 高：若直接把 `cli.py` 改写成 Web 服务，参数解析、REPL 控制流、HTTP 入口、会话状态会混在一起，后续难以维护。
- 高：若在第一阶段就做完整前后端分离、SPA、WebSocket 或流式推送，会把入口迁移放大成平台级重建。
- 高：若不先补 Web 等价测试，直接降级 `test_cli_roundtrip.py` 与 README，正式门禁会断层。
- 中：Web 会话若不以 `STM` 为唯一真相源，容易出现前端历史、后端快照双真相源冲突。
- 中：当前输入模型依赖 `goal / action_type / target / session_id` 等强约束，Web 若放宽输入会牵动 `clarify` 与错误映射。
- 中：若第一阶段就引入浏览器自动化，会破坏默认 `test_all.sh` 的轻量、可重复特性。
- 低：内核层 `contracts / runtime / adapters / knowledge / skills / orchestration / perception` 可直接复用。
- 低：前端第一阶段无需真正前端框架，使用单页薄界面即可平替当前 CLI/REPL 的最小交互能力。

## 执行顺序（Phase 1..N）
- Phase 1：抽共享入口服务层
  - 新增 `src/mz_agent/app/` 或 `src/mz_agent/entrypoints/`
  - 抽出 `runtime_factory`、`conversation_service`、结构化响应映射
- Phase 2：新增 Web API 薄入口
  - 新增 `src/mz_agent/web/app.py`
  - 新增 `src/mz_agent/web/schemas.py`
  - 最小 API：
    - `POST /api/round`
    - `GET /api/session/{session_id}/history`
    - `POST /api/session/{session_id}/reset`
    - `GET /api/session/{session_id}/status`
- Phase 3：补最小前端壳
  - 单页薄界面
  - 动作类型选择、目标输入、历史面板、状态栏、结果展示
  - 不做 WebSocket、多标签、富文本、复杂消息时间轴
- Phase 4：迁移测试门禁
  - 新增：
    - `test_web_roundtrip.py`
    - `test_web_session_resume.py`
    - `test_web_clarify_flow.py`
    - `test_web_error_mapping.py`
    - `test_web_live_config_guard.py`
    - `test_web_session_controls.py`
  - 保留现有内核回归
  - 让 Web 集成测试接管正式入口门禁
- Phase 5：迁移 README 与旧入口定位
  - 将 CLI/REPL 改为内部调试入口说明
  - 将 Web API / Web 页面改为主入口文案
  - 再决定 `cli.py`、`test_cli_roundtrip.py`、`test_repl_history.py` 是否移入 `distroy/`
