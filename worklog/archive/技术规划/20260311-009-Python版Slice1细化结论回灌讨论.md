## Team 分工（架构/实现/测试/风险）
- 架构：
  - 当前先只回灌 `Slice 1` 的第一项细化结论：5 个动态字段的最小类型、可空规则与默认值。
  - 目标是形成可直接写入主文档的最终矩阵，不再保留并行口径。
- 实现：
  - 采用 `Pydantic v2` 语义，明确区分“容器位”和“观察位”。
  - 需要把默认值写成显式结论，不能依赖类型注解隐式推断。
- 测试：
  - 后续要能直接围绕 `{}`、`None`、对象顶层类型、`extra='forbid'` 和 `model_dump` 写契约断言。
- 风险：
  - 最大风险是把 5 个字段全部做成宽松兜底容器，或把“空对象”和“缺席语义”混成同一件事。

## 共识结论（是否通过 + 条件）
- 结论：**通过。**
- 单结论：
  - **5 个动态字段最终收口为：4 个非空对象壳 + 1 个可空观察位。**
- 最终矩阵：

| 字段 | 最小类型 | 是否可空 | 默认值 | 回灌说明 |
|---|---|---:|---|---|
| `perception` | `dict[str, object]` | 否 | `{}` | 固定感知槽位，顶层必须是对象，使用 `default_factory=dict` |
| `stm` | `dict[str, object]` | 否 | `{}` | 固定短时记忆槽位，顶层必须是对象，使用 `default_factory=dict` |
| `skill_context` | `dict[str, object]` | 否 | `{}` | 固定技能上下文槽位，无上下文时也保留空对象，使用 `default_factory=dict` |
| `last_observation` | `dict[str, object] | None` | 是 | `None` | 唯一保留缺席语义的观察位，非空时顶层仍必须是对象 |
| `action_input` | `dict[str, object]` | 否 | `{}` | 动作唯一输入面，协议已冻结为缺省必须 `{}`，使用 `default_factory=dict` |

- 通过条件：
  - 5 个字段全部禁止裸 `Any`。
  - `perception / stm / skill_context / action_input` 不使用 `None` 表达缺省。
  - `last_observation` 是当前唯一允许 `None` 的动态字段。
  - 这一轮只锁顶层对象边界，不提前细化内部业务键结构。

## 风险清单（高/中/低）
- 高：若 `action_input` 允许 `None`，会直接冲突“缺省必须是 `{}`”这一冻结结论。
- 高：若 `perception / stm / skill_context` 放宽为可空，会把稳定运行态槽位重新做软。
- 高：若任一字段退化成裸 `Any`，Python `Slice 1` 的协议边界会立刻变软。
- 中：若把 `last_observation` 也默认成 `{}`，会丢失“当前没有最近观察”的缺席语义。
- 中：若直接用字面量 `{}` 而不是 `default_factory=dict`，会引入可变默认值风险。
- 低：`dict[str, object]` 仍偏宽，但对 `Slice 1` 是当前最稳的最小承载，不必在这一轮提前升级为递归 JSON 类型。

## `runtime/transitions.py` 纯函数契约

### Team 分工（架构/实现/测试/风险）
- 架构：
  - 4 个纯函数统一只承担状态门禁与版本失效，不承担对象修补或结果组装。
  - 目标是形成可直接回灌主文档的唯一契约，不再保留“返回 bool”或“塞进 validator”的并行解释。
- 实现：
  - 统一采用 keyword-only 签名。
  - 成功时返回目标状态或规范化后的新值，违规时抛领域异常。
- 测试：
  - 后续 `test_state_transitions.py` 只围绕“允许转换 / 禁止转换 / 版本失效 / 终止态阻断”写断言。
- 风险：
  - 最大风险不是函数名不对，而是把 `transitions` 写成修补器，或把结果映射与编排责任塞进来。

### 共识结论（是否通过 + 条件）
- 结论：**通过。**
- 单结论：
  - **`validate_step_transition / validate_cursor_transition / validate_react_transition` 统一采用 keyword-only 签名，成功返回目标状态，违规抛 `TransitionViolationError`。**
  - **`invalidate_current_step_on_plan_change` 在版本未变时返回原 `current_step`，版本变化时返回 `None`，不负责重推导新 step。**
  - **状态转移、版本失效、终止态阻断和 `clarify` 禁入主状态规则，全部禁止放进 Pydantic validator。**
- 最终契约矩阵：

| 函数 | 最小输入 | 最小输出 | 错误语义 | 回灌说明 |
|---|---|---|---|---|
| `validate_step_transition` | `current: StepState`，`target: StepState` | 返回 `StepState`，即通过校验后的 `target` | 非法转换抛 `TransitionViolationError` | 只判断 `step_state` 允许/禁止转换，不修对象 |
| `validate_cursor_transition` | `current: CursorState`，`target: CursorState` | 返回 `CursorState`，即通过校验后的 `target` | 非法转换抛 `TransitionViolationError` | 只判断游标状态转换，不处理计划逻辑 |
| `validate_react_transition` | `current: ReactStatus`，`target: ReactStatus` | 返回 `ReactStatus`，即通过校验后的 `target` | 非法转换抛 `TransitionViolationError` | 只判断运行态主状态转换，拒绝把 `clarify` 当主状态 |
| `invalidate_current_step_on_plan_change` | `previous_plan_version: str`，`current_plan_version: str`，`current_step: CurrentStep | None` | 版本未变返回原 `current_step`，版本变化返回 `None` | 版本变化本身不报错；仅非法输入才可视为契约违规 | 只负责让旧 `current_step` 失效，不负责重建新 step |

- 通过条件：
  - 4 个函数全部保持无副作用纯函数。
  - 不返回 `bool`，不返回“兼容通过”的模糊结果。
  - 不在函数内部做默认值填充、结果映射、编排调度或 `STM` 回写。

### 风险清单（高/中/低）
- 高：若 `validate_*` 返回 `bool`，调用方会各自补一层状态处理逻辑，接口看似统一、行为实际分裂。
- 高：若把状态矩阵、`clarify` 主状态禁入或版本失效塞进 Pydantic validator，模型层会越过 `Slice 1` 职责边界。
- 高：若把 Guardrails 映射、`ReActResult` 组装或新 `current_step` 推导塞进 `transitions`，门禁层会直接滑成编排层。
- 中：若 `invalidate_current_step_on_plan_change` 在版本变化时直接抛异常，会把正常的计划刷新错误建模成失败流。
- 中：若签名不用 keyword-only，`current` 与 `target` 更容易在调用点传反。
- 低：`plan_version` 当前先承载为 `str` 作为最小口径即可，后续若需细化再单独收口。

## 跨对象一致性真值表

### Team 分工（架构/实现/测试/风险）
- 架构：
  - 当前只收口 `plan_version / current_step / cursor_state / react_status / next_action / final_answer` 的跨对象联动。
  - 目标是形成可直接回灌主文档的真值表，不把结果组装和内容生成提前塞进 `Slice 1` 的门禁层。
- 实现：
  - 先区分三层语义：
    - 失效是协议语义
    - 清空是承载结果
    - 误用过期引用才是错误
- 测试：
  - `test_state_transitions.py` 主测跨版本失效、终止态清空和阻断态停点。
  - `test_guardrails_mapping.py` 主测决策到结果表达的唯一映射。
- 风险：
  - 最大风险是允许“半联动”状态存在，例如计划已换版但旧游标仍在、已终止但动作仍可继续推进。

### 共识结论（是否通过 + 条件）
- 结论：**通过。**
- 单结论：
  - **`plan_version` 变化时，旧 `current_step / cursor_state / NextAction` 先失效并在承载上清空；变化本身不报错，误用旧引用才报领域异常。**
  - **`react_status = finished` 时，`current_step` 必须置空，`next_action` 必须为空，`final_answer` 必须存在。**
  - **进入 `blocked / degraded` 后，本轮不得再推进任何新动作；当前主线仍采用最小表达：`next_action = None`、`final_answer = None`。**
- 通过条件：
  - `step_state` 始终保留为 Planning 权威字段，不由 ReAct / Guardrails / STM 直接改写。
  - “推导新的 `current_step`”仍属于后续编排责任，不在当前失效规则中顺带完成。

### `plan_version` 变化后的真值表

| 对象/行为 | 最终处理 | 回灌说明 |
|---|---|---|
| `current_step` | 必须立即失效，置为 `None` | 旧游标不能跨版本沿用 |
| `cursor_state` | 与旧 `current_step` 一起失效 | 不是改成别的合法状态，而是旧语义整体作废 |
| `step_state` | 只认新 `Plan.step_state` | 保持 Planning 权威，不从旧运行态迁移 |
| `NextAction` | 基于旧计划推导出的动作必须作废，置空或视为无效 | 旧动作上下文已经过期 |
| 是否直接报错 | 不因版本变化本身报错 | 版本变化是正常失效触发条件 |
| 何时报错 | 仅当版本已变后仍继续使用旧 `current_step / cursor_state / NextAction` | 错的是误用过期引用，不是发生版本变化 |

### `react_status = finished` 后的真值表

| 对象/行为 | 最终处理 | 回灌说明 |
|---|---|---|
| `current_step` | 必须置空 | 总表已冻结 |
| `next_action` | 必须为空 | 不能“已完成但仍有动作待执行” |
| `final_answer` | 必须存在有效终值 | `finished` 必须收束到完成性输出 |
| 是否允许继续推进新动作 | 不允许 | `finished` 是终止态 |
| `STM` 回写 | 允许，但只能在 `post_answer` 后发生 | 不得提前回写 |
| `step_state` | 不自动改写 | 仍由 Planning 权威面决定 |

### 进入 `blocked / degraded` 后的真值表

| 项目 | `blocked` | `degraded` | 共通要求 |
|---|---|---|---|
| `react_status` | `blocked` | `degraded` | 都属于本轮停点 |
| `next_action` | `None` | `None` | 都不再携带新动作 |
| `final_answer` | `None` | `None` | 当前 `Slice 1` 最小表达不通过 `final_answer` 承载，避免双轨 |
| 是否允许继续推进新动作 | 不允许 | 不允许 | 总表已冻结 |
| 是否允许继续外部动作链 | 不允许 | 不允许 | `tool / mcp / rag / llm / skill` 都应停止 |
| `current_step` | 保留当前值但冻结，不再推进 | 保留当前值但冻结，不再推进 | 便于保留现场与审计 |
| `cursor_state` | 不自动推进到 `completed` | 不自动推进到 `completed` | 阻断不等于完成 |
| `step_state` | 不自动改写 | 不自动改写 | 仍是 Planning 权威字段 |
| `STM` 回写 | 若发生，只能在本轮终点钩子后记录结果，不得借回写推进状态 | 同左 | 回写是记录，不是修补流程 |

### 测试归属口径
- `test_state_transitions.py`
  - `plan_version` 切换后的旧 `current_step / cursor_state` 失效
  - `react_status = finished` 后 `current_step` 清空
  - `finished / blocked / degraded` 后不得再推进新动作
  - 非法终止态回转与误用过期引用
- `test_guardrails_mapping.py`
  - `block / degrade / clarify` 到 `react_status / next_action` 的唯一映射
- `test_plan_refs.py`
  - 只负责对象结构，不负责跨版本和跨状态行为

### 风险清单（高/中/低）
- 高：若 `plan_version` 变化后仍保留旧 `NextAction`，会直接执行到错误计划分支。
- 高：若 `finished` 后仍保留动作型 `next_action`，终止态语义会立即失效。
- 高：若 `blocked / degraded` 后允许动作链继续推进，总表门禁就等于被放空。
- 中：若把 `step_state` 随运行态一起自动改写，会直接冲掉 Planning 权威边界。
- 中：若把 `blocked / degraded` 现在就通过 `final_answer` 做多轨表达，会和后续 Guardrails 最小映射冲突。
- 低：后续如果需要在阻断/降级场景补用户可见说明，应作为更高层输出策略单独讨论。

## Guardrails 映射真值表

### Team 分工（架构/实现/测试/风险）
- 架构：
  - 当前只收口 `allow / block / degrade / clarify / finish` 如何唯一落到 `ReActResult / NextAction / react_status / final_answer`。
  - 目标是形成单一结果面，不允许同一语义同时散落在状态、动作和额外标志位三处。
- 实现：
  - Guardrails 映射层只负责“把决策翻译成结果对象”，不负责状态机合法性判断。
  - `transitions` 只负责状态门禁，不负责结果翻译。
- 测试：
  - `test_guardrails_mapping.py` 主测“决策如何落到结果表达”。
  - `test_state_transitions.py` 仍只测“状态能不能这样变、停点后还能不能继续推进”。
- 风险：
  - 最大风险是把 `clarify` 做成主状态，或让 `block/degrade` 出现多轨表达。

### 共识结论（是否通过 + 条件）
- 结论：**通过。**
- 单结论：
  - **`allow` 不单独编码为新状态或新动作，只表示“保留原候选结果继续流转”。**
  - **`block / degrade` 在当前 `Slice 1` 最小结果面里只落 `react_status`，并统一 `next_action = None`、`final_answer = None`。**
  - **`clarify` 只体现为 `next_action.action_type = clarify`，不能成为主状态。**
  - **`finish` 只体现为 `react_status = finished + final_answer`，并统一 `next_action = None`。**
- 通过条件：
  - `block / degrade` 当前不通过 `final_answer` 承载，避免与后续更高层输出策略形成双轨。
  - `finish` 不是 Guardrails 决策值，而是回答类动作收口后的完成态表达。

### 最小结果矩阵

| 语义 | `react_status` | `next_action` | `final_answer` | 是否终止本轮 | 唯一表达 |
|---|---|---|---|---:|---|
| `allow + 动作候选` | `running` | 保留原动作 | `None` | 否 | `allow` 本身不单独落字段，只表示放行原候选结果 |
| `allow + 回答候选` | `finished` | `None` | 必须有值 | 是 | 允许原回答结果收束为完成态 |
| `block` | `blocked` | `None` | `None` | 是 | 只允许用 `react_status=blocked` 表达 |
| `degrade` | `degraded` | `None` | `None` | 是 | 只允许用 `react_status=degraded` 表达 |
| `clarify` | `running` | `action_type=clarify` | `None` | 否 | 只允许落在 `next_action.action_type` |
| `finish` | `finished` | `None` | 必须有值 | 是 | 只允许用 `react_status=finished + final_answer` 表达 |

### 字段级最终规则

#### `allow`
- 必须有：
  - `react_status=running`
- 可以有：
  - `next_action`
  - 或 `final_answer`
- 必须禁止：
  - `react_status=allow`
  - `next_action.action_type=allow`

#### `block`
- 必须有：
  - `react_status=blocked`
- 必须空：
  - `next_action`
  - `final_answer`
- 必须禁止：
  - `blocked + clarify`
  - `blocked + finish`
  - `blocked + 可执行动作`

#### `degrade`
- 必须有：
  - `react_status=degraded`
- 必须空：
  - `next_action`
  - `final_answer`
- 必须禁止：
  - `degraded + clarify`
  - `degraded + finish`
  - `degraded + 可执行动作`

#### `clarify`
- 必须有：
  - `next_action`
  - `next_action.action_type=clarify`
  - `react_status=running`
- 必须空：
  - `final_answer`
- 必须禁止：
  - `react_status=clarify`
  - `clarify + finished`
  - `clarify + blocked/degraded`

#### `finish`
- 必须有：
  - `react_status=finished`
  - `final_answer`
- 必须空：
  - `next_action`
- 必须禁止：
  - `finished + clarify`
  - `finished + 可执行动作`
  - 只有 `final_answer` 但没有 `finished`

### 测试归属口径
- `test_guardrails_mapping.py`
  - `allow` 不单独产出终止态
  - `block -> react_status = blocked`
  - `degrade -> react_status = degraded`
  - `clarify -> next_action.action_type = clarify`
  - `finish` 不属于 Guardrails 决策值
  - 禁止双轨表达
- `test_state_transitions.py`
  - `running -> blocked / degraded / finished` 的合法性
  - 进入 `blocked / degraded / finished` 后不得继续推进新动作
  - `clarify` 不得成为主状态

### 风险清单（高/中/低）
- 高：若把 `clarify` 同时做成动作类型和主状态，状态机与结果映射会立刻分叉。
- 高：若 `block / degrade` 允许保留动作型 `next_action`，Guardrails 门禁就等于失效。
- 高：若 `finish` 被混进 Guardrails 决策值，回答链和决策层会彻底混层。
- 中：若 `allow` 被编码成正式结果态，会平白多出一套状态语义。
- 中：若现在就让 `block / degrade` 通过 `final_answer` 承载用户可见说明，会和当前最小结果面形成双轨。
- 低：以后若需要补阻断/降级场景下的用户可见说明，应作为更高层输出策略单独讨论，不影响当前真值表。

### 执行顺序（Phase 1..N）
- Phase 1：先把这张 Guardrails 映射真值表回灌到 `Slice 1` 主文档。
- Phase 2：四组细化结论都回灌后，再统一判断是否进入首刀实现准备态。
