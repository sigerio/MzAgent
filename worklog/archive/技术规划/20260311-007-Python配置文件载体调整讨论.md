- 明确 Python 包元信息、测试配置、工具配置中哪些必须依赖标准载体，哪些可以迁入自定义 `.config`。
- 给出可执行的配置载体方案，但本轮不进入代码修改。

## Team 分工（架构/实现/测试/风险）
- 架构：判断 `pyproject.toml` 与 `.config` 各自承载哪些职责，避免配置层失稳。
- 实现：判断 Python package、依赖管理、安装入口是否还能脱离 `pyproject.toml`。
- 测试：判断 `pytest` 配置与测试发现机制是否能直接迁入 `.config`。
- 风险：避免因为替换标准配置载体，导致 Python 主线首刀无法被标准工具识别。

## 文档依据
- `pytest` 官方文档确认：
  - 支持 `pyproject.toml`
  - 支持 `pytest.toml`
  - 支持 `.pytest.toml`
  - 没有把任意 `.config` 作为标准配置入口
- `setuptools` 官方文档确认：
  - Python 项目元信息与依赖声明可以放在：
    - `pyproject.toml`
    - `setup.cfg`
    - `setup.py`
  - 官方示例没有把任意 `.config` 作为标准包元信息和构建入口

## 共识结论（是否通过 + 条件）
- 结论：本轮讨论**通过**。
- 单结论：
  - **`pyproject.toml` 可以取消。**
  - **但不能简单“改名成一个任意 `.config` 文件”并期待标准 Python 工具链自动识别。**
  - **如果你坚持不保留 `pyproject.toml`，更稳妥的方案是：标准工具配置改用 `setup.cfg + setup.py + pytest.toml`，而 `.config` 只承担项目内部配置职责。**
- 通过条件：
  - 构建与包元信息仍使用标准工具支持的载体。
  - 测试配置仍使用 `pytest` 官方支持的标准文件。
  - `.config` 不再承担构建工具入口职责，只承担内部运行配置职责。

## 为什么 `.config` 不能直接替代 `pyproject.toml`

### 1. 对 Python 包构建工具来说
- `setuptools` 官方文档给出的标准载体是：
  - `pyproject.toml`
  - `setup.cfg`
  - `setup.py`
- 没有官方支持“把项目元信息和构建入口写进任意 `.config` 并自动识别”的路径。

### 2. 对 pytest 来说
- `pytest` 官方文档支持：
  - `pyproject.toml`
  - `pytest.toml`
  - `.pytest.toml`
- 同样没有把任意 `.config` 作为 pytest 配置入口。

### 3. 所以新的问题不是“要不要改名”
- 真正的问题变成：
  - 哪些配置必须继续放标准载体
  - 哪些配置可以迁入内部 `.config`

## 推荐的新配置职责划分

### A. 标准包与构建配置
- 推荐使用：
  - `setup.cfg`
  - `setup.py`
- 负责：
  - 包名
  - 版本
  - Python 版本要求
  - 依赖
  - `src/` 布局包发现
- 原因：
  - 这是官方支持的 `pyproject.toml` 替代路线
  - 比强行让 `.config` 承担构建入口更稳

### B. 标准测试配置
- 推荐使用：
  - `pytest.toml`
  - 或 `.pytest.toml`
- 负责：
  - `testpaths`
  - `python_files`
  - 基础 `addopts`
- 原因：
  - 这是 `pytest` 官方支持的标准入口
  - 不需要再把测试配置绑回 `pyproject.toml`

### C. 项目内部配置
- 推荐使用：
  - `.config`
  - 或 `.config/` 目录
- 负责：
  - 项目自己的运行时约束
  - 非标准工具需要的内部开关
  - 未来的应用层配置
- 不负责：
  - Python 包元信息
  - 构建入口
  - pytest 官方配置入口

## 文件名改为 `mz.toml` 的复核（2026-03-11）

### 当前新增诉求
- 用户明确否决：
  - `setup.cfg + setup.py + pytest.toml + .config`
- 用户新要求：
  - **把 `pyproject.toml` 文件名改为 `mz.toml`**

### 共识结论（是否通过 + 条件）
- 结论：本轮讨论**不通过**。
- 单结论：
  - **`mz.toml` 可以作为项目内部自定义 TOML 配置文件存在。**
  - **但它不能直接替代 `pyproject.toml` 充当标准 Python 包元信息、构建入口和 pytest 标准配置入口。**
- 不通过原因：
  - `setuptools` 官方支持的标准载体仍是：
    - `pyproject.toml`
    - `setup.cfg`
    - `setup.py`
  - `pytest` 官方支持的标准载体仍是：
    - `pyproject.toml`
    - `pytest.toml`
    - `.pytest.toml`
  - 当前没有官方依据支持把：
    - `pyproject.toml`
    直接改名为：
    - `mz.toml`
    之后仍被标准工具自动识别。

### 哪些内容可以改成 `mz.toml`
- 项目内部运行配置
- 项目私有约束
- 将来应用层配置
- 非标准工具依赖的自定义配置

### 哪些内容不能只放在 `mz.toml`
- Python 包元信息
- 构建系统入口
- 依赖声明
- pytest 标准测试配置入口

### 风险清单（高/中/低）
- 高：若直接把 `pyproject.toml` 改名为 `mz.toml`，Python 标准构建与测试工具链将无法按标准入口识别项目。
- 高：若坚持只保留 `mz.toml`，后续会被迫引入非标准包装脚本，反而增加首刀复杂度。
- 中：若同时保留标准入口和 `mz.toml`，会增加一层配置同步成本，但这是当前唯一不破坏标准工具链的稳妥方案。
- 低：`mz.toml` 作为内部配置文件仍然有价值，只是不应承担标准入口职责。

### 当前可执行的两种合法方案
- 方案 A：
  - 保留 `pyproject.toml` 作为标准入口
  - 另增 `mz.toml` 作为项目内部配置
- 方案 B：
  - 放弃 `pyproject.toml`
  - 改走 `setup.cfg + setup.py`
  - `mz.toml` 仍只做内部配置

### 执行顺序（Phase 1..N）
- Phase 1：先明确你是在追求：
  - 统一文件名偏好
  - 还是完全替换标准入口
- Phase 2：若坚持标准工具链可直接工作，则不能只保留 `mz.toml`。
- Phase 3：只有在你确认接受方案 A 或方案 B 之一后，下一步的 Python 主线蓝图和 Slice 1 文档才能继续收敛。
