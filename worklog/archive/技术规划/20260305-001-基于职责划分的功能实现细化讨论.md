## 实现细化讨论（v1）

### 1) 感知与上下文构建（Perception）
- 最小实现：
  - `raw_input` 预处理（去噪、空值校验、长度限制）。
  - 结构化意图抽取（任务类型、约束、对象、优先级）。
  - 生成 `normalized_query` 与 `domain_tags`。
- 关键实现点：
  - 输入校验失败直接返回 `PER_001`，不进入下游模块。
  - 约束缺失时触发 `PER_003`，允许进入“澄清动作”而非硬失败。
  - 输出结构必须保持稳定字段顺序与类型，避免下游解析分歧。
- 依赖：
  - 短期记忆快照（可选）作为上下文补充。
  - Guardrails 对输入内容做敏感策略预检。

### 2) 短期记忆（Short-term Memory）
- 最小实现：
  - 事件流写入（`user_input/tool_result/model_output`）。
  - 窗口消息与摘要双轨存储（最近 N 条 + 历史摘要）。
  - `slot_state` 与 `token_budget_state` 实时更新。
- 关键实现点：
  - 采用会话级版本号避免并发写冲突（冲突返回 `STM_002`）。
  - 写入成功后才允许下游使用最新快照，确保读写一致性。
  - token 预算达到阈值时自动触发压缩策略。
- 依赖：
  - ReAct、Planning、LLM 都依赖该模块输出的快照。

### 3) LLM 接入层（多供应商统一网关）
- 最小实现：
  - 统一请求模型（消息、采样参数、工具 schema）。
  - 路由策略（质量优先/速度优先/成本优先）。
  - 统一响应标准化（文本、工具调用、用量、供应商追踪）。
- 关键实现点：
  - 超时、限流、不可达错误映射为 `LLM_002/003/001`。
  - 供应商失败时执行同策略降级重试（受重试预算约束）。
  - 所有供应商响应都转换为统一结构，禁止把供应商原始结构泄漏到上层。
- 依赖：
  - Guardrails 对模型输入/输出执行策略检查。

### 4) RAG（本地库 + 云端检索）
- 最小实现：
  - 查询改写 -> 多源召回 -> 重排 -> 证据打包。
  - 返回证据与引用信息，供回答与可追溯展示。
- 关键实现点：
  - `kb_scope` 驱动检索源选择，支持 local/cloud/both。
  - 当多源证据冲突且不可消解时返回 `RAG_004`。
  - 检索超时需区分“部分结果可用”和“完全失败”两类状态。
- 依赖：
  - LLM 可用于查询改写与冲突裁决（可选）。
  - ReAct 决定是否触发 RAG、触发次数与中止条件。

### 5) Tool / Function Calling
- 最小实现：
  - 工具注册表（名称、参数 schema、权限域）。
  - 参数校验与执行器调度。
  - 标准化 `tool_result` 与执行元信息返回。
- 关键实现点：
  - 参数不合法立即返回 `TOL_002`，不进入执行层。
  - 幂等工具支持重试，非幂等工具默认不自动重试。
  - 结果版本 `result_schema_version` 必须显式透传给上层。
- 依赖：
  - Guardrails 前置权限校验。
  - ReAct 决定工具调用时机与后续动作。

### 6) MCP 接入层
- 最小实现：
  - 能力发现（discover）与资源读写调用（read/call）。
  - 多传输协议封装（先支持最小集合）。
  - 统一返回 `capabilities` / `mcp_response` / `protocol_meta`。
- 关键实现点：
  - 协议版本不兼容返回 `MCP_002`，不可静默降级。
  - 服务不可达与参数错误区分处理（`MCP_001` / `MCP_003`）。
  - 能力缓存要有 TTL，避免每轮全量发现。
- 依赖：
  - Tool 层可复用 MCP 资源作为可调用能力。

### 7) Skills 系统（最小可用）
- 最小实现：
  - 技能元数据索引（意图、参数、适用条件）。
  - 单技能选择与执行（不做多技能编排）。
  - 返回技能执行计划与结果。
- 关键实现点：
  - 无匹配技能返回 `SKL_001`，由上层决定回退到通用推理。
  - 技能加载与执行错误分离（`SKL_002` / `SKL_003`）。
  - 技能参数映射必须显式声明，禁止隐式字段透传。
- 依赖：
  - ReAct 决策层作为技能调用入口。

### 8) ReAct 循环
- 最小实现：
  - `Think -> Act -> Observe` 循环骨架。
  - 步数上限与终止条件控制（`max_steps`）。
  - 动作结果回写短期记忆并驱动下一步。
- 关键实现点：
  - 步数超限返回 `RAC_001` 并触发降级回答策略。
  - 无可执行动作返回 `RAC_002`，需支持“请求澄清”分支。
  - `reasoning_trace` 默认裁剪，避免泄漏过多内部链路信息。
- 依赖：
  - Planning 提供步骤框架，ReAct负责执行层决策。

### 9) Planning（最小分解）
- 最小实现：
  - 目标拆解为顺序步骤（先不做并行与图优化）。
  - 步骤状态机维护（todo/in_progress/done/abandoned）。
  - 输出 `plan_id` 与步骤清单供 ReAct 执行。
- 关键实现点：
  - 计划不可执行返回 `PLN_003`，并附最小诊断信息。
  - 步骤依赖冲突返回 `PLN_002`，避免在 ReAct 层才暴露。
  - 计划更新必须保留版本号，支持中途重规划。
- 依赖：
  - 输入依赖 Perception 与短期记忆快照。

### 10) 安全与自我监控（Guardrails）
- 最小实现：
  - 前置检查（pre_action/pre_answer）。
  - 后置检查（post_action/post_answer）。
  - 输出 `decision`、`risk_level`、`risk_events`。
- 关键实现点：
  - `allow/block/degrade/clarify` 四类决策必须有清晰优先级。
  - 策略缺失返回 `GDR_001`，不允许默认为放行。
  - 误拦截（`GDR_003`）必须支持人工确认回路。
- 依赖：
  - 覆盖 Perception、LLM、Tool、MCP、ReAct 等关键节点。

## 跨模块收口要点
- 依赖方向建议：
  - Perception -> STM -> Planning -> ReAct -> (LLM/RAG/Tool/MCP/Skills) -> Guardrails -> STM。
- 冲突裁决建议：
  - ReAct 负责“执行动作选择”，Planning 负责“步骤框架约束”。
  - Tool/MCP/Skills 冲突时，优先显式技能路由，其次工具调用，最后回退通用 LLM。
- 状态一致性建议：
  - 所有模块共享 `request_id`、`session_id`、`plan_id(可选)`。
  - 每次关键动作后回写 STM，保证后续决策基于同一快照。

## 本轮新增结论（2026-03-05-Perception强依赖项）
- 按两套参考框架综合后，Perception 的强依赖可分为“必需依赖”与“条件依赖”。

### 必需依赖（不具备则 Perception 无法稳定工作）
- 输入消息标准化能力：
  - 需要统一消息对象与字段结构，保证输入可解析、可追踪。
  - 参考：AgentScope 的 `formatter` 抽象与 OpenAI 格式化实现。
- 会话上下文与短期记忆快照能力：
  - 需要读取最近对话、系统指令、历史摘要等上下文作为感知输入。
  - 参考：HelloAgents 的 `HistoryManager` / `ContextBuilder._gather`；AgentScope 的 `memory.add/get_memory`。
- Token 预算与上下文裁剪能力：
  - 需要在感知阶段控制上下文体量，避免后续推理输入失控。
  - 参考：HelloAgents 的 `_select`（预算填充）与 `_compress`；AgentScope 的 memory 压缩流程。
- 系统策略注入能力：
  - 需要在感知输出前合并系统约束（role/policy），作为标准化上下文的一部分。
  - 参考：HelloAgents `_gather/_structure`；AgentScope `sys_prompt + formatter.format`。

### 条件依赖（启用对应能力时变为强依赖）
- Query Rewrite 能力（启用 RAG 时）：
  - 用于把用户输入改写为可检索查询，直接影响感知后的检索质量。
  - 参考：AgentScope `_retrieve_from_knowledge` 中 `enable_rewrite_query` 流程。
- 检索能力（启用 RAG 时）：
  - 感知阶段需要把检索证据纳入上下文增强闭环。
  - 参考：AgentScope `_retrieve_from_knowledge`；HelloAgents `ContextBuilder` 预留了证据分区结构。
- 可观测链路（生产态）：
  - 需要记录感知输入、提取结果与错误码，便于回溯与调优。
  - 参考：HelloAgents `TraceLogger` 集成点。

## 本轮新增结论（2026-03-05-slot_state与token_budget_state必需性）
- `slot_state`：阶段性非必需，建议“字段保留、值可空”。
  - 当系统进入多轮澄清、参数收集、计划执行状态跟踪时，`slot_state` 会从可选变为必需。
  - 在当前“先独立模块讨论”阶段，可先不实现完整槽位系统，但应预留接口位置，避免后续破坏兼容。
- `token_budget_state`：工程上近必需，建议本阶段实现最小版本。
  - 即使不做复杂编排，也需要对上下文窗口、成本和压缩阈值做可观测控制。
  - 最小实现可只包含：`context_window`、`used_tokens`、`remaining_tokens`、`compression_threshold`。
- 阶段建议：
  - M0：`slot_state` 可空 + `token_budget_state` 最小可用。
  - M1：随 Perception/RAG 进入后补齐 `slot_state` 结构化语义与填充规则。

## 本轮冻结结论（2026-03-05-STM功能明确与移植策略）
- 用户确认：STM 参照 `src/` 现有框架移植核心功能。
- 约束确认：
  - `slot_state`：预留接口，当前可空；
  - `token_budget_state`：本阶段必须最小可用。

### 一) STM 本阶段目标（M0）
- 实现会话级消息存储与读取（追加、查询、清理）。
- 实现历史压缩能力（摘要 + 最近窗口保留）。
- 实现最小 token 预算状态跟踪。
- 保持与上层弱耦合，不引入编排与冲突裁决逻辑。

### 二) 核心移植映射（来源 -> MzAgent目标）
- HelloAgents `HistoryManager.append/get_history/compress`：
  - 移植为 STM 的事件追加、快照读取、摘要压缩三项基础能力。
- HelloAgents `TokenCounter` 与 `Agent._should_compress/_compress_history`：
  - 移植为 token 统计与压缩阈值触发机制。
- AgentScope `MemoryBase`（add/get_memory/delete_by_mark/update_messages_mark）：
  - 移植为统一 memory 接口层，方便后续替换存储后端。
- AgentScope `InMemoryMemory`：
  - 移植为默认后端（单进程、快速迭代）。
- AgentScope `AsyncSQLAlchemyMemory`（顺序索引 + 会话隔离）：
  - 作为下一阶段持久化实现参考，不要求本阶段落地数据库。

### 三) STM 接口冻结（M0）
- `append_event(session_id, event_type, payload, expected_version?) -> {version, seq}`
- `get_snapshot(session_id, window_size?) -> {memory_snapshot, slot_state, token_budget_state, version}`
- `compact_summary(session_id, strategy) -> {version, summary_text}`
- `clear_session(session_id) -> {cleared_count}`

### 四) 字段冻结（M0）
- `slot_state`：
  - 类型：`object | null`
  - 约束：默认可空，预留字段，不做业务语义填充。
- `token_budget_state`（最小可用）：
  - `context_window`：总预算上限
  - `used_tokens`：当前已使用
  - `remaining_tokens`：剩余额度
  - `compression_threshold`：压缩触发阈值（比例或绝对值）

### 五) 错误码冻结（M0）
- `STM_001`：会话不存在。
- `STM_002`：版本冲突（并发写入或 `expected_version` 不匹配）。
- `STM_003`：写入失败或存储异常。

### 六) 实施边界（本次不做）
- 不实现跨模块调用顺序、功能编排、冲突裁决逻辑。
- 不实现完整槽位语义系统（仅占位）。
- 不强制接入数据库（默认 InMemory；持久化后续阶段评估）。

## 本轮新增结论（2026-03-05-STM风险清单与解决方案）

### 风险 1（高）：并发写入导致会话状态不一致
- 现象：
  - 多次写入交错后，事件顺序或快照版本出现覆盖/回退。
- 解决方案：
  - 引入 `expected_version` 乐观并发控制，版本不匹配返回 `STM_002`。
  - `append_event` 与版本递增采用原子写入（单事务/单临界区）。
  - 写冲突后统一“重读快照再重试”，不做隐式自动合并。

### 风险 2（中）：token 预算不可观测导致压缩失控
- 现象：
  - 上下文超窗后才被动失败，无法提前触发压缩。
- 解决方案：
  - 落地 `token_budget_state` 最小四字段：`context_window`、`used_tokens`、`remaining_tokens`、`compression_threshold`。
  - 每次 `append_event` 后增量更新 `used_tokens/remaining_tokens`。
  - 达到阈值时触发 `compact_summary`，压缩后回写预算状态。

### 风险 3（中）：未预留槽位接口导致后续协议破坏
- 现象：
  - 后续接入 Perception/Planning 时不得不改动 STM 响应结构。
- 解决方案：
  - 在 `get_snapshot` 固定返回 `slot_state` 字段（当前允许 `null`）。
  - 约定 `slot_state` 仅可“由空到有”扩展，禁止字段语义反转。
  - 在 M1 阶段新增槽位语义时，保持 M0 兼容。

### 风险 4（低）：仅 InMemory 导致重启后会话丢失
- 现象：
  - 进程重启后历史上下文无法恢复。
- 解决方案：
  - M0 接受该限制，作为阶段性取舍。
  - 预留存储抽象接口（对齐 `add/get/clear/mark`），后续平滑切换持久化后端。
  - M1 评估 SQLite/SQLAlchemy 持久化方案，优先实现会话级恢复。

  - 修补：策略版本化 + 原子切换；上线前执行 `dry_run` 差异检查。

### 七) 本次不做
- 不实现复杂策略 DSL 解释器（M0 仅支持基础规则格式）。
- 不实现自动学习型策略更新。
- 不实现跨会话全局风控画像。

## 本轮新增结论（2026-03-05-Guardrails参考依据核查）
- 结论：`src` 下有可直接借鉴的 Guardrails 组件，但均为“分散能力”，尚无单一完整 Guardrails 引擎实现。

### A. HelloAgents 侧可借鉴点
- 工具访问控制：
  - `ToolFilter` 抽象与 `ReadOnly/FullAccess/Custom` 三类过滤器，可直接映射 `allow/block` 前置判定。
- 工具熔断机制：
  - `CircuitBreaker` 提供连续失败熔断、恢复超时与状态查询，可映射 `degrade/block` 执行保护。
- 工具执行接入：
  - `ToolRegistry.execute_tool` 在执行前检查熔断，执行后记录结果，可作为 Guardrails 与 Tool 层联动参考。

### B. AgentScope 侧可借鉴点
- 工具中间件拦截链：
  - `Toolkit.register_middleware` + `_apply_middlewares` 提供 onion 模式前后置拦截，可直接映射 Guardrails 的 `pre_action/post_action`。
- 统一错误与降级出口：
  - `call_tool_function` 对不存在工具、未激活工具、MCP 异常均统一转为 `ToolResponse`，可借鉴“稳定返回而非抛散异常”策略。
- MCP 内容规范化：
  - `_convert_mcp_content_to_as_blocks` 提供多模态统一转换，可借鉴 Guardrails 输出规范化前的内容标准化步骤。

### C. 对 MzAgent Guardrails 的落地建议
- 以 HelloAgents 的“过滤 + 熔断”作为规则执行动作库；
- 以 AgentScope 的“中间件拦截链 + 统一响应”作为执行框架；
- 在其上补齐本项目的 `decision/risk_level/risk_events/rule_hits` 标准协议。

## 本轮新增结论（2026-03-05-Guardrails协议影响程度评估）
- 结论：`decision / risk_level / risk_events / rule_hits` 对 Guardrails 的影响程度为**极高（核心级）**，属于“先定义再实现”的基础协议。

### 影响分级
- 功能正确性影响：高（约 40%）
  - `decision` 直接决定放行、拦截、降级、澄清路径；
  - 无该字段将无法形成可执行判定闭环。
- 风险治理影响：高（约 30%）
  - `risk_level` 与 `risk_events` 决定风险分层与响应动作；
  - 缺失会导致高风险与低风险同处置，治理失真。
- 可观测与审计影响：中高（约 20%）
  - `rule_hits` 是“为何命中”的可追溯依据；
  - 无法追踪命中规则将难以复盘误拦截与策略漂移。
- 上下游兼容影响：中（约 10%）
  - 上层模块依赖统一判定结构做后续处理；
  - 协议不稳定会引发接口抖动与重复适配。

### 若未冻结该协议的直接后果
- 无法稳定区分 `block` 与 `degrade`，行为不可预测。
- 风险事件不可追溯，难以定位误判来源。
- 规则迭代后难以比较前后差异，回归成本显著上升。
- 后续接入 Tool/MCP/LLM 时需重复改接口，返工高。

## 本轮冻结结论（2026-03-05-Guardrails-M0冻结清单）

### 一) 字段字典（M0 冻结）
- 顶层返回：
  - `status`：`ok | error`
  - `error_code`：`"" | GDR_001 | GDR_002 | GDR_003`
  - `message`：判定摘要
  - `latency_ms`：判定耗时
- 核心判定：
  - `decision`：`allow | block | degrade | clarify`
  - `risk_level`：`low | medium | high`
  - `rule_hits`：命中规则 ID 列表（按命中顺序）
  - `risk_events`：风险事件数组（可空）
- `risk_events[*]` 结构：
  - `event_id`：事件唯一 ID
  - `rule_id`：命中规则 ID
  - `severity`：`low | medium | high`
  - `message`：事件说明
  - `evidence`：证据片段（可脱敏）
  - `stage`：`pre_action | post_action | pre_answer | post_answer`

### 二) 判定矩阵（M0 冻结）
- 决策优先级（全局固定）：
  - `block > degrade > clarify > allow`
- 阶段判定规则（最小版）：
  - `pre_action`：
    - 命中高危规则 -> `block`
    - 命中中危可替代规则 -> `degrade`
    - 约束不充分 -> `clarify`
    - 其余 -> `allow`
  - `post_action`：
    - 检测到越权结果/敏感泄露 -> `block`
    - 结果可修剪后继续 -> `degrade`
    - 结果语义不确定 -> `clarify`
    - 其余 -> `allow`
  - `pre_answer`：
    - 命中合规禁区 -> `block`
    - 可降级为安全答复模板 -> `degrade`
    - 信息不足需追问 -> `clarify`
    - 其余 -> `allow`
  - `post_answer`：
    - 抽检命中严重风险 -> `block`（并记录事件）
    - 可回写修订建议 -> `degrade`
    - 边界不清 -> `clarify`
    - 其余 -> `allow`

### 三) 错误码对照（M0 冻结）
- `GDR_001`（策略缺失）：
  - 触发：策略包未加载/版本不可用/关键规则组缺失
  - 默认决策：`block`
  - 必填事件：`rule_id=policy_missing`
- `GDR_002`（策略执行异常）：
  - 触发：规则引擎运行时异常、表达式求值失败
  - 默认决策：`block`
  - 必填事件：`rule_id=engine_runtime_error`
- `GDR_003`（疑似误拦截）：
  - 触发：命中冲突规则且置信不足，或历史上高误判规则再次命中
  - 默认决策：`clarify`
  - 必填事件：`rule_id=potential_false_positive`

### 四) 兼容与演进约束（M0 冻结）
- 字段兼容：
  - M0 字段只允许“新增可选字段”，禁止删除或改语义。
- 枚举兼容：
  - `decision/risk_level` 仅可追加，不可重定义既有值。
- 审计兼容：
  - `rule_hits` 与 `risk_events` 在 `status=ok` 时也允许为空数组，但字段必须存在。
