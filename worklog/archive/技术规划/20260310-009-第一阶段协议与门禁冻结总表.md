- 结论：本轮讨论**通过**，该总表已经足以作为第一阶段实现前的协议与门禁冻结依据。

- 当前判断：
  - 可以进入子仓落地蓝图与受控实现切片讨论。
  - 仍不允许跳过门禁直接展开多模块自由编码。

## 参考框架边界

### 可借力承载点
- `src/HelloAgents`
  - 可借力：轻量 `ReAct` 循环、`ToolRegistry`、`SkillLoader`、`SessionStore`、上下文打包壳。
- `src/agentscope`
  - 可借力：`ReActAgent` 装配壳、`Toolkit`、`Memory`、`MCP`、`PlanNotebook`、hook 与 middleware 挂点。

### 不可直接照搬的部分
- `src` 里的对象模型、计划模型、hook 机制和运行时字段，不是 MzAgent 的权威协议。
- `PlanNotebook` 更偏运行中计划工具，不能直接替代第一阶段 `Planning M0`。
- `ThinkingBlock`、多模态块、细粒度流式字段等运行时细节，不进入第一阶段最小契约。

## 第一阶段最小状态转移

### 状态集合
- `step_state`：`todo`、`in_progress`、`done`、`abandoned`
- `react_status`：`running`、`finished`、`blocked`、`degraded`
- `cursor_state`：`pending`、`active`、`completed`、`abandoned`
- `guardrails_decision`：`allow`、`block`、`degrade`、`clarify`

### 允许转换
- `step_state`
  - `todo -> in_progress`
  - `in_progress -> done`
  - `todo -> abandoned`
  - `in_progress -> abandoned`
- `cursor_state`
  - `pending -> active`
  - `active -> completed`
  - `pending -> abandoned`
  - `active -> abandoned`
- `react_status`
  - `running -> running`
  - `running -> finished`
  - `running -> blocked`
  - `running -> degraded`

### 必须禁止的转换
- `step_state`
  - 禁止 `todo -> done`
  - 禁止 `done -> in_progress`
  - 禁止 `done -> todo`
  - 禁止 `abandoned -> todo`
  - 禁止 `abandoned -> in_progress`
- `cursor_state`
  - 禁止 `pending -> completed`
  - 禁止 `completed -> active`
  - 禁止 `completed -> pending`
  - 禁止 `abandoned -> pending`
  - 禁止 `abandoned -> active`
- `react_status`
  - 禁止 `finished -> blocked`
  - 禁止 `finished -> degraded`
  - 禁止 `blocked -> finished`
  - 禁止 `blocked -> degraded`
  - 禁止 `degraded -> finished`
  - 禁止 `degraded -> blocked`
  - 禁止把 `clarify` 当作新的主状态

### 跨层权威边界
- `Planning` 只产出 `step_state`，不产出 `current_step`
- `ReAct` 负责推导 `current_step`
- `STM` 负责持久化运行态
- `Guardrails` 不得直接改写计划结构，也不得直接回写 `step_state / cursor_state`
- `plan_version` 一旦变化，旧 `current_step` 与旧 `cursor_state` 必须立即失效

## 第一阶段阻断级契约测试断言

### 对象契约组
- `ContextSnapshot` 顶层必须固定包含：`perception`、`stm`、`current_plan`、`skill_context`、`last_observation`
- `CurrentPlanRef` 必须固定包含：`plan_id`、`plan_version`、`current_step`、`step_state`
- `CurrentStep` 若存在，必须同时包含：`step_id`、`cursor_state`
- `AvailableAction.targets` 缺值时必须是 `[]`
- `NextAction.action_input` 缺值时必须是 `{}`
- `ExecutionContext` 必须固定包含：`request_id`、`session_id`、`plan_id`

### 权威来源组
- `Planning` 输出只允许包含：`plan_id`、`plan_version`、`steps`、`step_state`
- `Planning` 不得产出：`current_step`、`tool_choice`、`execution_hint`
- `current_step` 只能由 `ReAct` 推导、由 `STM` 持久化

### 执行链门禁组
- `skill / tool / mcp / rag / llm` 只能走动作链：`pre_action -> execute -> post_action`
- `clarify / finish` 只能走回答链：`pre_answer -> output -> post_answer`
- Guardrails 决策映射必须唯一：
  - `allow` 不产生终止态
  - `block -> react_status = blocked`
  - `degrade -> react_status = degraded`
  - `clarify` 只体现为 `next_action.action_type = clarify`
- `STM` 回写只能发生在 `post_action` 或 `post_answer` 之后

### 状态转移组
- 规划层只允许：
  - `todo -> in_progress`
  - `in_progress -> done`
  - `todo -> abandoned`
  - `in_progress -> abandoned`
- 游标层只允许：
  - `pending -> active`
  - `active -> completed`
  - `pending -> abandoned`
  - `active -> abandoned`
- `plan_version` 一旦变化：
  - 旧 `current_step` 必须立即失效
  - 旧 `cursor_state` 必须立即失效
  - 必须基于新 `step_state` 重新推导 `current_step`
- `react_status = finished` 时：
  - `current_step` 必须置空
- 进入 `blocked` 或 `degraded` 后：
  - 本轮不得再推进新的动作

## 当前不进入第一阶段阻断范围的内容
- 多 provider 路由细节、Anthropic/Gemini 专属字段
- 多模态内容块细节、流式 `is_last` 一类运行时字段
- 复杂审批流、动态策略热更新、完整审计平台字段
- 非主链字段的 Tool/MCP/LLM 扩展属性

## 风险清单（高/中/低）
- 高：若对象契约与状态转移不在子仓先落硬约束，协议漂移几乎必然发生。
- 高：若 `current_step` 权威来源不锁死，`Planning / ReAct / STM` 会各写各的。
- 高：若 `STM` 回写时机不锁死，运行态会在未过 Guardrails 决策前被提前污染。
- 中：若空值表达不统一，schema 与实现会出现多套风格。
- 低：后置字段长期不补，会增加第二阶段扩展返工成本。

## 执行顺序（Phase 1..N）
- Phase 1：先在 `sub/MzAgentSub` 落对象定义、状态校验和契约测试骨架。
- Phase 2：再以本总表为依据接入最小编排链：`Planning -> ReAct -> Guardrails -> STM`。
- Phase 3：最后逐个接入 `tool / mcp / rag / llm / skill` 适配层。

## Python 主线承载说明（2026-03-11）

### 共识结论（是否通过 + 条件）
- 结论：本轮讨论**通过**，本总表继续保留为第一阶段唯一语言无关协议与门禁依据，但其首版承载方式已从 Rust 主线切换为 Python 主线。
- 单结论：
  - **总表里的主协议语义、状态机语义、权威边界和阻断级断言继续沿用。**
  - **需要替换的不是“冻结结论”，而是“结论落在哪种技术载体上”。**
- 通过条件：
  - Python 主线下仍不得让 `HelloAgents` 或 `agentscope` 的现成对象反向定义本总表。
  - Python 主线下仍坚持“先门禁、后功能”。

### Python 主线推荐承载方式
- 对象契约：
  - 使用 `Pydantic` 模型承载
- 状态值域：
  - 使用 `Enum`
- 状态门禁：
  - 使用独立纯函数校验
- 契约测试：
  - 使用 `pytest`
  - 入口保持在 `tests/contracts/`

### Python 主线下不变的内容
- `ContextSnapshot / CurrentPlanRef / CurrentStep / ExecutionContext / NextAction / ReActResult / GuardrailDecision` 的职责边界不变
- `step_state / cursor_state / react_status / guardrails_decision` 的状态集合不变
- `Planning` 不产出 `current_step`、`ReAct` 推导 `current_step`、`STM` 持久化运行态这一权威边界不变
- `skill / tool / mcp / rag / llm` 只能走动作链、`clarify / finish` 只能走回答链这一分流规则不变
- `STM` 只能在 `post_action / post_answer` 后回写这一门禁不变

### Python 主线下替换掉的内容
- Rust 的：
  - `Cargo.toml`
  - `src/lib.rs`
  - `mod.rs`
  - `serde / serde_json / thiserror`
  - trait 壳与 crate 切片表达
- 需要被 Python 的：
  - `pyproject.toml`
  - `src/mz_agent/...`
  - `Pydantic + Enum + pytest`
  承载方案替代

### Python 主线下的执行顺序
- Phase 1：先在 `sub/MzAgentSub` 落 Python 版对象定义、状态校验和契约测试骨架。
- Phase 2：再落 Python 版 `trace / writeback` 门禁。
- Phase 3：最后接 Python 版主链编排骨架与适配层。
