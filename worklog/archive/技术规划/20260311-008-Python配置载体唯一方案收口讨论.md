## Team 分工（架构/实现/测试/风险）
- 架构：

- 实现：

- 测试：
  - `pytest` 官方标准入口支持 `pyproject.toml / pytest.toml / .pytest.toml`。
  - 为了让 `tests/contracts/` 稳定发现，测试配置可以独立于包元信息，但不应反向制造主线分叉。
- 风险：
  - 一旦让 `pyproject.toml`、`setup.cfg + setup.py`、`mz.toml`、`.config` 同时作为主线候选，最先返工的会是包元信息、依赖声明、测试发现和后续 CI 接入。

## 共识结论（是否通过 + 条件）
- 结论：**通过。**
- 单结论：
  - **Python 主线下的唯一正式配置方案，收口为：保留 `pyproject.toml` 作为唯一标准入口。**
  - **`mz.toml` 或 `.config` 若未来需要存在，只能作为项目内部运行配置，不得承担构建、依赖声明和 pytest 标准入口职责。**
- 通过依据：
  - 已复核的官方事实：
    - `pytest` 标准配置入口支持：
      - `pyproject.toml`
      - `pytest.toml`
      - `.pytest.toml`
    - `setuptools` 标准构建入口围绕：
      - `pyproject.toml`
  - 因此，`pyproject.toml` 是当前唯一同时满足“标准构建入口 + 标准测试入口 + 与现有 Python 主线蓝图一致”的单文件主方案。
- 当前不通过的方案：
  - 直接用 `mz.toml` 替代 `pyproject.toml`
  - 直接用任意 `.config` 替代标准入口
  - 在当前阶段默认改走 `setup.cfg + setup.py + pytest.toml`
  - 让标准入口和自定义入口并列承担同一职责
- 通过条件：
  - `Slice 1` 阶段只保留一个标准入口，不再并行保留替代方案。
  - 后续所有 Python 主线文档都以 `pyproject.toml` 为唯一正式入口回灌统一。
  - 若将来出现内部运行配置需求，再单开议题讨论 `mz.toml` 或 `.config`，且不得触碰标准工具链入口职责。

## 风险清单（高/中/低）
- 高：若继续保留 `pyproject.toml` 与 `mz.toml` / `.config` / `setup.cfg + setup.py` 并行候选，主线会再次分叉。
- 高：若让非标准命名文件承担标准入口职责，Python 构建与 `pytest` 发现会直接失去标准识别路径。
- 高：若现在默认切到 `setup.cfg + setup.py + pytest.toml`，会直接偏离当前 Python 主线蓝图和 `Slice 1` 文档，制造文档与实现双返工。
- 中：若未来确实再增加内部运行配置文件，职责边界若不单独锁死，仍可能产生二次配置漂移。
- 低：`pytest` 配置将来是否拆到 `pytest.toml`，属于后续可选优化项，不影响当前唯一方案的成立。
