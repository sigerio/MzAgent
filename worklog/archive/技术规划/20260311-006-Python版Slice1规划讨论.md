## 文档依据
- Pydantic v2 官方文档确认：
  - `BaseModel` 可作为主数据模型载体
  - `model_validate` 可作为输入校验入口
  - `model_dump` 可作为稳定导出接口
  - `ConfigDict(extra='forbid')` 可直接禁止多余字段
  - `Enum` 可直接作为字段类型参与校验与序列化
- pytest 官方文档确认：
  - 可在 `pyproject.toml` 中使用 `[tool.pytest.ini_options]`
  - 可在其中声明 `testpaths`
  - 默认测试文件命名可以继续采用 `test_*.py`
  - 首批普通断言风格即可，不需要先引入额外断言插件

## 共识结论（是否通过 + 条件）
- 结论：本轮讨论**通过**，Python 版 `Slice 1` 已足以收敛为一个最小但可直接落地的首刀规划。
- 单结论：
  - **`Slice 1` 的对象层统一使用 `Pydantic BaseModel + ConfigDict(extra='forbid')`。**
  - **状态层统一使用 `Enum`。**
  - **状态门禁统一使用 `runtime/transitions.py` 中的独立纯函数。**
  - **首批测试统一使用 `pytest`，并集中在 `tests/contracts/`。**
- 通过条件：
  - `Slice 1` 不引入 `orchestration`、`adapters`、`trace`、`writeback`。
  - `Slice 1` 不引入 provider、MCP transport、CLI、配置加载器等依赖。
  - 动态字段必须有明确的受控承载方式，不能直接退化成无边界 `Any`。

## Pydantic 承载基线

### 统一模型基线
- `contracts/*` 下的对象模型统一：
  - 继承 `BaseModel`
  - 显式声明字段
  - 默认使用 `model_validate`
  - 对外导出时统一使用 `model_dump`
- 推荐基础配置：
  - `model_config = ConfigDict(extra='forbid')`
- 目的：
  - 防止“顺手多塞字段”导致协议漂移

### 动态字段承载策略
- 当前这些字段必须允许结构化但受控的动态值：
  - `perception`
  - `stm`
  - `skill_context`
  - `last_observation`
  - `action_input`
- 推荐首刀策略：
  - 先使用 `dict[str, object] | None` 这类最小受控承载
  - 不直接放成裸 `Any`
  - 不在 `Slice 1` 里过早细化成大量嵌套子模型
- 原因：
  - 要兼顾首刀速度
  - 但不能完全失去边界

### 状态与决策承载
- `StepState`
- `CursorState`
- `ReactStatus`
- `GuardrailsDecisionKind`
- 统一使用：
  - `Enum`
- 不使用：
  - 普通字符串常量散落在各模块

## runtime 层基线

### `runtime/errors.py`
- 首批只收敛：
  - `ContractError`
  - `TransitionError`
  - `ValidationError`（领域层，不与 Pydantic 同名导出混淆时需谨慎命名）
- 更稳的命名建议：
  - `ContractViolationError`
  - `TransitionViolationError`
  - `DomainValidationError`
- 目标：
  - 与 Pydantic 的错误面区分开

### `runtime/transitions.py`
- 必须保留独立纯函数，不把状态门禁塞进模型 validator：
  - `validate_step_transition`
  - `validate_cursor_transition`
  - `validate_react_transition`
  - `invalidate_current_step_on_plan_change`
- 原因：
  - 这样 `state_transitions` 测试才能保持门禁测试而不是流程测试

## 首批测试门禁

### `test_state_transitions.py`
- 锁：

  - 允许转换
  - 禁止转换
  - `clarify` 不得成为主状态
  - `plan_version` 变化后的旧游标失效出口

### `test_context_snapshot.py`
- 锁：
  - `ContextSnapshot` 顶层字段固定
  - `ExecutionContext` 顶层追踪字段固定
  - 多余字段在 `extra='forbid'` 下被拒绝

### `test_plan_refs.py`
- 锁：
  - `Plan / PlanStep / CurrentPlanRef / CurrentStep` 结构固定
  - `CurrentStep` 不脱离 `CurrentPlanRef` 语义漂移

### `test_guardrails_mapping.py`
- 锁：
  - `AvailableAction.targets` 缺省必须是数组
  - `NextAction.action_input` 缺省必须是对象
  - `GuardrailDecision` 到 `NextAction / ReActResult` 有唯一承载面
  - `clarify` 只体现为动作类型

## 风险清单（高/中/低）
- 高：如果 `contracts/*` 不统一启用 `extra='forbid'`，Python 下协议漂移会比 Rust 更快暴露。
- 高：如果把状态门禁写进 Pydantic validator 而不是独立纯函数，测试会很快退化成对象校验混合测试。
- 高：如果动态字段直接放成裸 `Any`，首刀虽然快，但会马上失去协议边界。
- 中：如果 `runtime/errors.py` 直接复用 Pydantic 的 `ValidationError` 语义，领域错误和模型错误会混在一起。
- 中：如果首刀就引入 `pytest-asyncio`、provider SDK 或 tracing 依赖，讨论会立刻滑向 Slice 2/3。
- 低：后续可能需要把动态字段再细化成子模型，但当前先保持受控动态承载更稳。
