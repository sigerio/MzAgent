# Web 端 UI TODO 清单梳理

- 记录时间：2026-03-14
- 用户原始诉求：梳理 TODO 清单
- 当前状态：讨论中（未进入代码修改）。

## 本轮讨论目标
- 基于已冻结的 Web UI 方案与 UI 任务编排，整理一份更细粒度的 TODO 清单。
- 让后续进入代码修改阶段时可以直接按 TODO 顺序推进，而无需再次拆分。
- 区分“首版必须项”“建议补强项”“延期项”，避免实现范围失控。

## 当前已知约束
- 本轮属于项目问题，需先记录并进入讨论。
- 未获得用户明确同意前，不执行代码修改。
- 首版 UI 已冻结为“单页任务控制台”，不是类聊天产品。
- 前端形态已冻结为 `单页薄界面 + 原生表单/轻量 JS`。
- 会话真相源必须仍为 `STM`，前端只维护瞬时展示状态。
- 首版门禁已冻结为：`基础提交流`、`会话续写流`、`clarify/错误映射流`、`重置流`。

## Team 分工（架构/实现/测试/风险）
- 架构：把 UI 编排阶段进一步拆成可执行 TODO，并明确依赖关系。
- 实现：细化页面区块、交互、状态、会话控制与样式收口项。
- 测试：把首版 UI 门禁映射到具体 TODO，避免测试悬空。
- 风险：把膨胀项和延期项单列，防止 TODO 污染首版边界。

## 共识结论（是否通过 + 条件）
- 结论：通过，可以形成首版 Web UI 的详细 TODO 清单。
- 通过条件：
  - TODO 必须沿用既定 7 个 Phase，不得跨阶段插入增强项。
  - TODO 要覆盖 UI 首版闭环，但不应提前进入完整产品化能力。
  - 每个 TODO 都要能映射到页面区块、API 契约或测试门禁中的至少一个对象。

## 风险清单（高/中/低）
- 高：
  - 若 TODO 清单把增强项与首版闭环混写，会导致执行阶段失去边界。
  - 若 TODO 不绑定 API 契约，会在实现阶段产生前后端语义漂移。
- 中：
  - 若 TODO 粒度过粗，执行时仍需二次拆解，降低推进效率。
  - 若 TODO 粒度过细，会把讨论文档变成实现流水账，影响维护。
- 低：
  - 样式与文案细节可后续再微调，不影响首版主链。

## 执行顺序（Phase 1..N）
- Phase 1：冻结依赖与映射关系。
- Phase 2：完成页面骨架 TODO。
- Phase 3：完成表单闭环 TODO。
- Phase 4：完成结果与历史 TODO。
- Phase 5：完成状态与会话控制 TODO。
- Phase 6：完成响应式与可用性 TODO。
- Phase 7：完成测试与文档 TODO。
