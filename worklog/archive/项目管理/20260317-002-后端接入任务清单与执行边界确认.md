## Team 分工（架构/实现/测试/风险）

- 架构：区分“已存在后端接口”和“当前仅在 UI 中预留、但后端未实现的接口”。
- 实现：把仍待完成的后端接入任务按模块拆为 `capabilities / retry / streaming / 深化能力扩展`。
- 测试：识别哪些应补单模块测试，哪些应补 Web 集成测试，避免只靠 `integration` 混测。
- 风险：识别“用户说完成所有后端接入”与“仓库内实际边界”之间的范围漂移风险。

## 共识结论（是否通过 + 条件）

- 结论：通过，可以形成当前“后端接入任务清单”。
- 条件：
  - 只把仓库中仍未完成、且有明确证据指向的后端接入项列入本轮。
  - 不把历史已完成的 `LLM / MCP / 最小 Web API` 误判为待办。
  - 不把纯前端任务混入后端接入执行清单。

## 风险清单（高/中/低）

- 高：`完成所有后端接入` 这句话范围过大；若不先收敛，会把“未来演进项”与“当前 UI 配套接口”一起开工，导致边界失控。
- 高：当前 active 仍有 `全量UI重构`，若直接改后端，容易与前端预留接口产生契约漂移。
- 中：`integration` 历史上多次被提醒不能替代单模块测试；若后端扩展只补集成测试，容易留下模块级空洞。
- 中：`skills / knowledge / perception` 的“真实能力扩展”是演进项，不一定都属于当前 Web 控制台必须阻断的后端接入。
- 低：历史 README 版本号与 active 文档版本记录可能不同步，执行时需以子仓当前文件真实状态为准。

## 执行顺序（Phase 1..N）

- Phase 1：确认本轮“后端接入”只包含当前 Web 控制台直接依赖的后端缺口。
- Phase 2：优先处理已被 UI 文档明确声明但后端未落地的接口。
- Phase 3：补对应单元/集成测试，锁住接口契约。
- Phase 4：再评估 `knowledge / skills / perception / integration` 是否进入下一轮独立演进任务。
