# cunzhi 结果未正常回流 LLM 收口

## 用户原始问题
发起团队讨论，cunzhi返回的内容没有正常告知LLM进行返回，仅显示在对话框中

## 当前讨论目标
- 分析 `cunzhi` MCP 返回内容为何只显示在对话框，而没有继续进入 LLM 收口链路。
- 判断问题位于路由决策、MCP 执行结果写回、还是 `respond_with_llm` / `pending_external_interaction` 控制逻辑。
- 在不修改代码的前提下，给出团队共识、风险清单和建议执行顺序。

## 初步观察
- 该问题直接涉及当前仓库 `cunzhi` MCP 与 Web/auto 编排逻辑，属于项目问题，需先记录并讨论，不能直接改代码。
- 仓库内已有多份 `cunzhi` 相关归档与 active 记录，本次问题更聚焦“`cunzhi` 结果未继续喂给 LLM 收口”的链路表现，因此单独建档。

## 团队讨论结论

### Team 分工（架构/实现/测试/风险）
- 架构：确认问题根因更偏向 `respond_with_llm / expect_user_followup` 的路由契约门禁，而不是 `MCP` 结果写回丢失。
- 实现：核查 `auto` 主流程在 `mcp` 返回后何时直接停在 `_complete_round_with_observation(...)`，何时继续进入 `_complete_round_with_llm(...)`。
- 测试：核查现有集成测试是否覆盖“`cunzhi` 成功后同轮继续喂给 LLM 收口”这一关键路径。
- 风险：评估若继续保留当前“先显示 `mcp` 文本、下轮再回 LLM”的行为，会造成哪些用户感知偏差与回归风险。

### 共识结论（是否通过 + 条件）
- 结论：通过。
- 当前现象主因不是 `cunzhi` 结果没有写入系统，而是当前 `auto` 契约在以下两类条件下会刻意终止同轮 LLM 收口：
  - `expect_user_followup == true` 且本轮执行结果来源是 `mcp`
  - `respond_with_llm == false`
- 具体表现：
  - 命中上述条件时，流程会直接走 `_complete_round_with_observation(...)`，返回 `result_type = "mcp"`、`llm_completion_enforced = false`，因此前端只看到 `cunzhi` 文本显示在对话框。
  - 只有不命中这两个门禁时，才会进入 `_complete_round_with_llm(...)`，把 `intermediate_observations` 里的 `mcp` 结果喂给 LLM 生成最终答复。
  - `pending_external_interaction` 的职责是为下一轮保留待续交互上下文，不等于“本轮一定继续走 LLM 收口”。
- 条件：
  - 如果产品要求“`cunzhi` 返回后同轮也必须由 LLM 再组织输出”，则需要调整门禁优先级，至少保证 `respond_with_llm = true` 时不要被 `expect_user_followup = true` 提前截断。
  - 如果产品要求“外部能力本轮就是先向用户追问”，那当前表现属于按契约执行，不应把这类场景统一改成同轮 LLM 收口。

### 风险清单（高/中/低）
- 高：当前视觉表现很容易被用户理解成“`cunzhi` 返回内容没有继续告知 LLM”，但实际上是契约层刻意不收口，容易反复被报为缺陷。
- 高：现有测试没有锁住“`mcp` 成功 + `respond_with_llm = true` + 同轮 LLM 收口”的正向路径，后续即使改了逻辑也容易再次回归。
- 中：`pending_external_interaction` 依赖 `observation.result.text` 提取，若 `cunzhi` 返回结构变化但仍是有效结果，待续上下文可能静默退化。
- 中：若直接粗暴去掉 `expect_user_followup` 的早返回，会破坏真正的“继续向用户追问”场景。
- 低：当前已有测试覆盖“跨轮回流到 LLM”，但这不足以证明“同轮也会回流”，容易形成错误安全感。

### 执行顺序（Phase 1..N）
- Phase 1：先确认产品契约，区分“追问型 `cunzhi`”与“总结型 `cunzhi`”是否都要求同轮 LLM 收口。
- Phase 2：若要求同轮收口，优先调整 `respond_with_llm / expect_user_followup` 的分支优先级，而不是先改写回存储层。
- Phase 3：补集成测试，至少覆盖 `action_type = mcp + expect_user_followup = true + respond_with_llm = true` 的成功路径。
- Phase 4：再评估是否需要放宽 `pending_external_interaction` 的结构依赖，避免 `result.text` 缺失时静默丢上下文。
