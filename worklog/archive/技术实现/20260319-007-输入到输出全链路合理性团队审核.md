# 输入到输出全链路合理性团队审核

## 用户原始问题
发起团队审核，基于此问题，审核整个输入到输出线路的合理性

## 当前讨论目标
- 基于当前 `cunzhi` 收口问题，审查 Web 端从用户输入到最终输出展示的整条链路是否合理。
- 覆盖范围包括：前端提交与本地渲染、`/api/round` 接口、`ConversationService` 自动编排、外部能力执行、写回历史、LLM 收口、SSE/快照同步与最终前端展示。
- 在不修改代码的前提下，给出团队审核结论、主要风险和建议执行顺序。

## 初步观察
- 该问题直接涉及当前仓库的主对话链路和产品交互语义，属于项目问题，需先记录并讨论，不能直接改代码。
- 本次审核是在 `20260319-006-cunzhi结果未正常回流llm收口.md` 基础上的扩展，从单点问题上升到全链路合理性审视，因此单独建档。

## 团队审核结论

### Team 分工（架构/实现/测试/风险）
- 架构：审核从前端输入、Web API、`ConversationService`、`auto` 路由到 LLM 收口的职责边界是否清晰，是否存在结构性语义混叠。
- 实现：审核真实控制流和数据流是否闭环，重点核查 optimistic render、同步 `/api/round`、`mcp`/`llm` 分流、writeback 和最终 history 对齐。
- 测试：审核现有测试对主链成功/失败、同轮/跨轮收口、SSE 快照和前端可见性是否覆盖充分。
- 风险：审核用户感知风险、协议语义风险与回归风险是否已被当前设计合理控制。

### 共识结论（是否通过 + 条件）
- 结论：有条件通过。
- 当前输入到输出主链在工程上是闭环可运行的，基本链路如下：
  - 前端提交时先做本地 optimistic render，并立即清空输入框；失败时回滚 history、状态和输入内容。
  - Web 层将 `POST /api/round` 同步映射到 `ConversationService.submit_round(...)`，接口语义是“本轮完成后回包”。
  - 后端通过 `prepare_snapshot_for_round(...)` 先挂起 `pending_user_message`，再进入 `auto` 路由、外部能力执行、最后在 `observation` 与 `llm` 两种收口路径中二选一。
  - 正式历史写入发生在 writeback 阶段，而不是提交瞬间；前端与后端因此天然存在一个“乐观展示 vs 最终落盘”的双轨时序。
- 但全链路的产品语义存在一处核心不稳定点：
  - 路由模型输出不仅决定“调用哪个动作”，还决定“本轮是否继续回到 LLM 收口”，即 `respond_with_llm / expect_user_followup` 同时承担动作选择与收口策略控制。
  - 在 `expect_user_followup == true && source == "mcp"` 或 `respond_with_llm == false` 时，链路会刻意停在 `_complete_round_with_observation(...)`，不再同轮进入 LLM 收口。
  - 因此当前系统会出现“技术上链路没断，产品上却像是只显示 MCP 结果、没有继续回答”的稳定分支，不是偶发异常。
- 条件：
  - 如果产品契约允许“追问型外部能力本轮只显示观察结果、下一轮再回 LLM”，当前设计整体可接受，但必须把该语义明确化。
  - 如果产品契约要求“外部能力成功后同轮必须给出 LLM 最终组织答复”，当前设计不够合理，需要调整 `respond_with_llm / expect_user_followup` 的优先级和测试覆盖。

### 风险清单（高/中/低）
- 高：用户感知风险。前端先显示/先清空、后端同步收口落盘之间存在时间差，用户容易误判“消息是否真正完成处理”。
- 高：协议语义风险。`respond_with_llm` 与 `expect_user_followup` 组合直接决定是否同轮 LLM 收口，这个决策隐藏在路由结果里，用户和前端都难以直观看懂。
- 高：测试缺口风险。缺少“`mcp` 成功 + `respond_with_llm = true` + 同轮 LLM 收口”的关键正向测试，未来改动后容易出现行为变化但测试仍全绿。
- 中：中间态可见性风险。`/api/round` 是同步一次性返回，SSE 又只是当前快照，不提供 round 执行中的增量事件，导致链路处理过程偏黑箱。
- 中：上下文连续性风险。`pending_external_interaction` 强依赖 `observation.result.text`，外部能力返回结构一旦变化，待续上下文可能静默退化。
- 中：前端状态一致性风险。当前 history 防回退策略依赖长度和末条元信息，在复杂并发场景下可能屏蔽合法更新。
- 低：目前主链 API 级测试覆盖尚可，能保障大部分后端分支可达，但不足以保障真实浏览器交互体验。

### 执行顺序（Phase 1..N）
- Phase 1：先冻结产品契约，明确区分“追问型外部交互”和“总结型外部交互”是否都要求同轮 LLM 收口。
- Phase 2：若要求同轮收口，优先调整 `ConversationService._execute_auto_round_record(...)` 中 `respond_with_llm / expect_user_followup` 的分支优先级，而不是先动写回层。
- Phase 3：补关键契约测试，至少覆盖：
  - `mcp` 成功 + `respond_with_llm = true` + 同轮 LLM 收口
  - 前端 optimistic render、提交清空、失败回填、history 回滚
  - SSE 与 `/api/round` 并发下的最终一致性
- Phase 4：再评估是否需要把 `/api/round` 从“同步整轮收口”升级为“增量事件流 + 最终状态对齐”，以降低用户感知黑箱。
