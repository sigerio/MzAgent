# LLM 连接测试按钮

- 记录时间：2026-03-18
- 用户原始诉求：添加一个测试当前LLM连接的按钮，用于检测当前LLM的配置是不是好的
- 当前状态：已完成“测试当前选中配置方案”功能实现，前端按钮与后端测试接口均已接入，并完成设置抽屉页面预览。

## 背景现状

- 当前 Web 配置页已有方案的保存、切换、删除能力，但没有独立的 LLM 连接测试入口。
- 当前后端已有配置方案读写接口：`/api/llm/profiles`
- 当前后端没有专门的“连接测试 / 探活”接口。
- 当前真实 LLM 调用由 `LLMAdapter` 负责，真实请求走 `client.responses.create(...)`。

## 本轮讨论目标

1. 明确“测试连接”功能应放在哪一层实现。
2. 给出最小可行实现路径。
3. 识别对现有配置流和运行时的影响。

## 初步判断

- 该需求应采用“前端按钮 + 后端专用测试接口”的方式实现。
- 不建议前端直接复用正式对话提交流程去试探配置是否有效。
- 最合理的做法是新增一个轻量测试接口，使用当前表单中的临时配置或当前已保存配置，向目标 LLM 发起一次最小请求，并返回结构化测试结果。

## 已完成实现

- 设置抽屉新增“测试连接”按钮，仅测试当前选中的已保存配置方案。
- 前端新增结果提示区，展示成功/失败信息、模型、耗时与返回摘要。
- 后端新增配置方案连接测试接口，路径为 `POST /api/llm/profiles/{name}/test`。
- 探活请求不复用正式对话链路，不写入会话历史，也不污染当前 session 状态。
- 连接测试错误已区分为缺少模型、缺少密钥、缺少 base_url、认证失败、超时、连接失败、限流等场景。
- 子仓版本已从 `0.1.21` 提升到 `0.1.22`。
- 已完成本地页面预览，截图路径：`/tmp/mzagent-llm-test-button.png`

## 本轮说明

- 当前仅做了页面接入与本地 UI 预览，未实际触发远端 LLM 探活请求，避免误用现有配置产生真实调用成本。
