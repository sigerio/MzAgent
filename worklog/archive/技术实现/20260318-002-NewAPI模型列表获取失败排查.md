# NewAPI 模型列表获取失败排查

- 记录时间：2026-03-18
- 用户原始诉求：没有办法从newapi处获取模型列表
- 当前状态：已进入实现与排查并行阶段，已落地部分代码修复，待真实站点复验。

## 背景现状

- 当前项目已切换为通过 NewAPI 网关统一管理连接。
- 当前“获取模型”实现路径为：
  - 用户保存 `base_url + api_key`
  - 后端请求 `{base_url}/models`
  - 解析返回的 OpenAI 风格 `data[].id`
- 用户反馈当前无法从 NewAPI 站点获取模型列表。

## 初步排查焦点

1. `base_url` 是否被用户填写为站点根地址，而不是带 `/v1` 的 API Base URL。
2. 当前实现是否应自动兼容 `base_url` 缺少 `/v1` 的场景。
3. 当前错误提示是否不足以定位 NewAPI 返回的真实问题。

## 当前排查结论

- 已确认首个真实问题不是 NewAPI 路径，而是 Web 入口路由遗漏：
  - `MzAgentWebApp.__call__` 只分发了 `/api/llm/profiles*`
  - 新增的 `/api/llm/connection` 与 `/api/llm/connection/models` 根本没有进入服务层
  - 这会直接导致前端“保存连接 / 获取模型”链路不可用
- 已确认第二个真实问题是空配置启动异常：
  - 在“当前无配置方案”时，`ConversationService -> build_runtime -> MCPAdapter`
  - `MCPAdapter` 初始化会提前调用 `load_runtime_settings`
  - 因为没有激活模型而直接抛错，破坏空态页面与新建连接流程

## 本轮已执行修复

- 已修复 Web 路由分发，`/api/llm/connection*` 现在会进入 LLM 路由处理器。
- 已修复空配置启动问题，无模型时页面仍可正常初始化并进入连接配置流程。

## 当前状态

- 本地最小验证已通过：
  - `POST /api/llm/connection` 返回 `200`
  - `POST /api/llm/connection/models` 已进入服务层，不再是 `404`
  - `GET /api/llm/connection/models` 已兼容接入，不再返回“`不支持的配置方案操作`”
- 由于当前沙箱网络受限，项目内最小验证访问外部站点时返回：
  - `获取模型失败：[Errno 1] Operation not permitted`
- 因此当前状态为：
  - 代码级链路已修复
  - 待用户在真实 NewAPI 站点上复验

## 参考仓调研补充

- 用户补充诉求：`src/new-api` 与 `src/opencode` 已提供，需参考别人如何处理“模型列表获取 / 自定义 Provider”。

### new-api 调研结论

- `new-api` 的公开推理入口 `GET /v1/models` 不是匿名发现接口，而是受令牌鉴权保护：
  - 路由位置：`src/new-api/router/relay-router.go`
  - 中间件：`middleware.TokenAuth()`
- `TokenAuth()` 会先把请求头中的认证信息归一到 `Authorization: Bearer <token>`，再执行令牌校验：
  - 位置：`src/new-api/middleware/auth.go`
- `GET /v1/models` 的返回内容是“当前令牌可见的模型列表”，不是站点全部模型：
  - 位置：`src/new-api/controller/model.go -> ListModels`
- `403` 在 `new-api` 内通常来自以下访问限制：
  - 令牌 IP 白名单不匹配
  - 用户已被封禁
  - 令牌指定了用户无权访问的分组
  - 分组已弃用
  - 普通用户试图通过 token 指定渠道
- 这说明当前用户遇到的 `403`，语义上更接近“令牌无权访问 `/v1/models` 返回的模型视图”，而不是“接口方法错误”。

### opencode 调研结论

- `opencode` 的 custom provider 方案不是“输入 base_url 后自动从远端拉模型列表”：
  - 表单要求用户手动填写 `models[]`
  - 位置：`src/opencode/packages/app/src/components/dialog-custom-provider-form.ts`
  - E2E 也只验证“手工添加模型 / 删除模型”，没有“远端发现模型”步骤
  - 位置：`src/opencode/packages/app/e2e/settings/settings-providers.spec.ts`
- `opencode` 的 provider 体系大量基础模型元数据来自 `models.dev`，不是来自用户填写的自定义 OpenAI 兼容站点：
  - 位置：`src/opencode/packages/opencode/src/provider/models.ts`
- `opencode` 官方文档明确说明：
  - OpenAI 兼容提供商默认使用 `@ai-sdk/openai-compatible`，适用于 `/v1/chat/completions`
  - 如果模型走 `/v1/responses`，则应使用 `@ai-sdk/openai`
  - 同一 provider 混用时，可在模型级设置 `provider.npm` 覆盖默认值
  - 文档位置：`src/opencode/packages/web/src/content/docs/zh-cn/providers.mdx`
- 运行时也确实按“provider 默认 + model 覆盖”来选择协议实现：
  - 位置：`src/opencode/packages/opencode/src/provider/provider.ts`

### 对当前问题的启发

- `new-api`：
  - `/v1/models` 可调用，但它受令牌权限控制，返回的是“这个 key 当前能看到什么”
  - 所以若用户站点返回 `403`，前端自动发现模型列表不能视为稳定能力
- `opencode`：
  - 对自定义 OpenAI-Compatible provider，主路径是“手工维护模型列表”
  - 协议差异通过 provider/model 配置显式声明，而不是依赖模型发现接口自动猜测

## 本轮执行修改

- 已按用户要求先对齐 `new-api TokenAuth` 语义调整模型发现请求：

## 真实环境复验结论

- 已直接基于项目内真实配置，对真实 NewAPI 站点执行 `discover_connection_models()`。
- 首次真实复验返回的并非令牌权限错误，而是 Cloudflare 拦截：
  - `status=403`
  - `error_code=1010`
  - `error_name=browser_signature_banned`
  - 含义：站点按客户端指纹封禁了当前模型发现请求
- 进一步探测确认：
  - 当模型发现请求显式带 `User-Agent` 时，站点可正常返回模型列表
  - 因此根因不是 `base_url`、不是 `api_key`、不是 `TokenAuth` 逻辑本身，而是请求头缺少可接受的 `User-Agent`
- 已落地修复：
  - 模型发现请求统一显式携带 `User-Agent: MzAgent/0.1.24 (+https://github.com/sigerio/MzAgentSub)`
- 修复后的真实环境结果：
  - 成功获取 `7` 个模型
