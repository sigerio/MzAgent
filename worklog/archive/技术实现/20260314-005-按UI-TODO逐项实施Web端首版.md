# 按 UI TODO 逐项实施 Web 端首版

- 记录时间：2026-03-14
- 用户原始诉求：逐一完成 TODO 清单
- 当前状态：实施中（已获得执行指令）。

## 本轮实施目标
- 按既定 Web UI TODO 清单逐项落地首版 Web 端实现。
- 实施范围限定为“单页任务控制台 + 轻量 Web API + 最小门禁测试”。
- 不引入多会话列表、流式输出、富文本消息、复杂调试面板等延期项。

## 当前已知约束
- 本轮属于项目问题，已进入代码修改阶段。
- 代码主要落点在 `sub/MzAgentSub` 子仓库。
- 按规则，本轮进入项目代码修改时需先执行子仓版本号 patch `+1`。
- Web 第一阶段仍以 `STM` 作为唯一会话真相源，前端不维护第二套历史状态。
- UI 形态冻结为 `单页薄界面 + 原生表单/轻量 JS`。

## Team 分工（架构/实现/测试/风险）
- 架构：先抽共享入口服务层，再接 Web API，最后接薄界面。
- 实现：按“服务层 -> API -> 页面 -> 文档”顺序逐项落地。
- 测试：补齐 Web 首版四类门禁流，不做像素级视觉回归。
- 风险：严格隔离延期项，避免 UI 首版膨胀成完整产品重构。

## 共识结论（是否通过 + 条件）
- 结论：通过，按既定 TODO 顺序进入实施。
- 条件：
  - 先处理依赖层和共享服务层，不直接把 `cli.py` 改写成 Web 服务。
  - Web 端所有会话操作都必须映射到真实 `STM`/服务层能力。
  - 新增测试需覆盖首版最小闭环。

## 风险清单（高/中/低）
- 高：若共享服务层没有先抽离，CLI 与 Web 逻辑会缠在一起。
- 高：若前端自行拼历史真相源，会破坏 `STM` 边界。
- 中：若 UI 先行、测试滞后，回归问题会在末端集中暴露。
- 低：视觉微调与帮助文案可以后续再收口。

- Phase 3：实现最小 Web API。
